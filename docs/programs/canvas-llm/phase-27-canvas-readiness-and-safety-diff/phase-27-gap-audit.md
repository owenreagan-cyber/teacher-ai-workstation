# Phase 27 Gap Audit (Phase 27B recovery)

Audit date: 2026-07-11. Scope: PR #323 (`938171b` "Phase 27 canvas readiness and safety diff", merged to `main` at `8df047e`).

Method: every row below is backed by direct inspection of the file/function named in
"File/function evidence." No row is based on the existence of a file name alone. Where the
current implementation returns a hard-coded or fixture-forced result, the exact line is quoted.

## Summary

The merged Phase 27 surface is six Python files (~180 lines total), one fixture file, three
four-line documentation stubs, one flat UI panel, a fully hard-coded status script, and a test
file that contains one real behavioral assertion block and one no-op `echo`. It is an honest
**scaffold**, consistent with its own `implementation-report.md` ("deterministic safety diff
**scaffolding**"), but the roadmap and status script currently present it as delivered
capability. Almost every subsystem required by the Phase 27 specification (matching precedence,
dependency graph, SQLite ledger, approval system, freshness, health checks, rollback planning,
transport boundary beyond a single rejection check, pagination, rate limiting, module placement,
drift detection, export package) is **missing**, not merely partial.

## Bottom line

Phase 27 as merged is a real, working **narrow slice** (canonicalization + a single comparison
function + a hard-coded manifest shape) wrapped in validators that cannot fail. The scope gap
between what PR #323 delivered and the full Phase 27 specification is large: of the ~40
subsystems enumerated in the mission's Part 4 checklist, roughly 4 are genuinely implemented
(canonicalization, the narrow field-diff, the manifest *shape*, CLI `build`), 2 are misleading
(the four hash "domains," the manifest's freshness/validation fields), and the remaining ~34 are
either missing outright or represented only by print statements. This is consistent with, and
confirms, the concern that prompted this audit.
